## Page 4 — 序章：好奇心驱动的三次转向

[type: content]

### 1.0 序章：好奇心驱动的三次转向

> **引用框**: 在进入互联网之前，我已经完成了三次"清零重来"。每一次转向的背后，都是同一个模式：**极度热爱 + 大力出奇迹**。

1. **高中 → 物理竞赛**
   - 对物理着迷，自学本科课程 — 五门普通物理（力学、热学、电磁学、光学、原子物理）+ 三门高数（微积分、线性代数、概率统计）
   - 刷完比新华词典还厚的《物理学大题典》，市面上所有能买到的竞赛题基本刷遍
   - 获奖保送

2. **本科 → 电子学**
   - 制定个性化学习方案，提前一年三年本科毕业
   - 国内第一批人研究量子通信与量子计算
   - 发现对物理的热情消退，不喜欢学术界的工作
   - 研究生转向更偏向应用的领域 — 电子工程

3. **研究生 → 代码和算法**
   - 在电子与物理的交叉领域做研究，发了几篇还不错的文章
   - 发现自己真正感兴趣的是代码和算法
   - 从零开始自学编程和算法，《C++ Primer》&《算法导论》课程和所有习题刷完，LeetCode 刷了三遍
   - 进入互联网行业

> Speaker Notes: 我的职业起点不是计算机科班，而是物理。高中时对物理着迷，自学竞赛保送；本科制定个性化学习方案提前一年三年毕业，当时是国内第一批研究量子通信与量子计算的人，但后来发现对物理的热情消退，也不喜欢学术界的工作方式，研究生就转向了更偏应用的电子工程方向。在电子与物理的交叉领域做研究，发了几篇还不错的文章，但又发现自己真正感兴趣的是代码和算法，于是从零开始自学编程和算法。每一次转向的逻辑都一样：不是"这条路走不通"，而是"我发现自己对另一个方向更着迷"。方法也一样 — 把所有能找到的材料一页一页啃完、一题一题刷完。看起来很笨，但这种笨功夫建立的认知深度是不可替代的。

## Page 5 — 三次转向的底层逻辑

[type: content]

### 为什么先讲这个故事

1. **极度热爱是起点** — 每次转向都不是"走不通"，而是"我发现自己对另一个方向更着迷"
2. **大力出奇迹是方法** — 没有捷径，一页一页啃完、一题一题刷完。看起来很笨，但这种**笨功夫建立的认知深度不可替代**
3. **从零开始的自学能力是可以训练的** — 物理→电子→代码，每次转向都在缩短"从零到能产出"的时间
4. **AI 时代最需要这种能力** — 技术每 2 年迭代一轮，"敢于清零、快速重建"比"在一个领域死磕"更重要

> **引用框**: 今天 Claude Code 可以让你几小时上手新技术栈。但**决定"要不要转向""往哪个方向转"的判断力**，AI 帮不了你。好奇心 + 判断力，是 AI 时代最稀缺的组合。

> Speaker Notes: 为什么要先讲这个故事？因为这三次转向的模式，和后面进入职场之后的每一次决策都是一脉相承的。极度热爱是起点 — 当你真正热爱一个东西时，投入不需要意志力驱动。大力出奇迹是方法 — 没有捷径。从零开始的自学能力是可以训练的 — 每转一次，"从零到能产出"的时间就更短。AI 时代最需要的就是这种能力 — 当技术每两年迭代一轮，"敢于清零、快速重建"比"在一个领域死磕"更重要。好奇心加判断力，是 AI 时代最稀缺的组合。这些转向背后有没有犹豫？当然有。每次都有。离开百度时犹豫过——刚入行一年就走？去腾讯时犹豫过——放弃创业公司的成就？来 Shopee 时犹豫过——又要从零开始？但我有一个判断方法：想象五年后的自己，如果会后悔"当时为什么没试"，那就去试。如果五年后的自己会无所谓，那就不急。关于"如何发现热爱"——说实话不是每个人都需要找到废寝忘食的热爱。更实际的方法是大量接触不同事物，留意哪些事让你进入心流状态——做的时候忘记了时间。不需要找到终身热爱，先找到当前最让你好奇的方向就够了。关于"从零开始的能力是学习能力还是心理韧性"——两者都是，但心理韧性更关键。学习能力可以靠方法论提升，心理韧性需要一次次"从不确定中活下来"才能建立。每次清零的前几周都很痛苦——你从专家变回新手，这种身份落差很折磨人。但经历几次之后你会发现"新手"不是可怕的身份，而是充满可能性的状态。试错建议：控制试错成本。不要用辞职去验证一个想法——先用业余时间做最小可行版本。AI 时代这更容易了，用 Claude Code 几个小时就能搭一个 prototype 来验证。

## Page 8 — 1.2 关键转折点

[type: content]

### 关键转折点：从"会调模型"到"模型团队负责人"

> **引用框**: 模型能力只是入场券，真正决定成败的是对数据和业务的深度理解。

- 2018 年做 DNN 精排模型上线，**第一版效果不好，AB 实验甚至是负向的**
- 团队里有声音说"还是老模型稳"
- **没有放弃** — 花了两周逐条排查 bad case
- 发现不是模型结构的问题，而是**样本构造和特征对齐**出了偏差
- 修复之后，指标大幅正向，最终推全
- 从这个项目之后，团队开始把更多的排序模块交给我负责

**最大的成长**：

- 从"给我一个任务我做好"变成**"我自己定义问题、设计方案、推动落地、验证结果"的全闭环能力**

**最深的教训**：

- **对线上系统要有敬畏心，任何改动都要有回滚方案** — AI 写的代码也需要灰度、监控和回滚机制

> Speaker Notes: 2018 年做 DNN 精排模型上线时，第一版效果不好，AB 实验甚至是负向的。团队里有声音说"还是老模型稳"。我没有放弃，花了两周逐条排查 bad case，发现不是模型结构的问题，而是样本构造和特征对齐出了偏差。修复之后指标大幅正向，最终推全。这件事让我意识到：模型能力只是入场券，真正决定成败的是对数据和业务的深度理解。也有教训 — online learning 时因为实时脏样本导致线上模型 train 崩，让我刻骨铭心地记住：对线上系统要有敬畏心。这个教训在 AI 时代同样适用。

## Page 11 — 1.3 从技术到管理的不适应

[type: content]

### 从技术到管理 — 最不适应的事

> **引用框**: 如果所有事情都需要你亲自做，那你就是团队的瓶颈，而不是团队的杠杆。管理者的价值不是"我能做多少事"，而是"我能让多少人做对事"。

- 做技术时，成就感来源很直接 — 模型指标涨了、系统稳定了、论文发了
- 做管理后，你的产出变成了**别人的产出**
- 最不适应的是：你明明知道这个问题自己上手半天就能解决，但你得**忍住，花两天教会团队成员去做**

**能力补短板**：

| 补充能力 | 具体实践 |
|----------|----------|
| 业务思考力 | 带领业务技术小组，从业务问题出发设计技术方案 |
| 团队管理 | 带领算法中心 + 产研中心，完成目标拆解、流程规范、成员培养、氛围建设 |
| 业务引领 | 任业务负责人，带领算法 + 产研团队完成竞品调研、业务拆解、方案落地 |

> Speaker Notes: 从技术到管理最不适应的是什么？做技术时成就感很直接 — 指标涨了、系统稳定了。做管理后，你的产出变成了别人的产出。最难受的是，你明明知道这个问题自己半天就能解决，但你得忍住，花两天教会团队成员去做。后来想通了：如果所有事情都需要你亲自做，那你就是团队的瓶颈，而不是团队的杠杆。管理者的价值不是"我能做多少事"，而是"我能让多少人做对事"。

## Page 12 — 1.3 技术引领：管理转型不等于放弃技术

[type: content]

### 技术引领 — 管理转型并不意味着放弃技术深度

1. **RL 广告混排** — 落地基于 BCQ+BatchRL 的强化学习广告混排框架，通过 Double Dueling DQN 和体验约束 loss 持续迭代，创新性被业界顶会 KDD 收录认可
2. **Trinity 召回** — 用一套聚类+直方图框架统一了多兴趣/长期兴趣/长尾兴趣三个独立问题，实现"三位一体"
3. **满意度优化体系** — 质疑推荐范式本身（拟合行为概率 ≠ 用户真实偏好），串联 RL 融合优化 → 长期满意度建模 → 偏好度蒸馏泛化学习

> **引用框**: 管理和技术不是二选一 — 技术深度为管理决策提供了更好的判断力。

## Page 24 — 2.2 故事回溯：Trinity — 把三个问题变成一个框架

[type: content]

### 故事回溯 — Trinity："认识本质"的更高层应用

**背景**：在腾讯做兴趣点发掘时，团队同时在做三个独立项目 — 多兴趣建模、长期兴趣建模、长尾兴趣探索。三个小组各自迭代，进展缓慢

**顿悟时刻**：

- 退一步看，它们本质上是**同一个问题** — 用户偏好在 cluster 空间上的分布
- 多兴趣 = 取直方图 top-N cluster（兴趣度最高的 N 个方向）
- 长尾兴趣 = 热度消偏后选取时间间隔最大的 cluster（稀有但有兴趣的方向）
- 长期兴趣 = 老种子 + 对老行为更"公平"的 embedding（避免遗忘早期兴趣）
- **一套聚类 + 一个直方图 + 定制化策略 = 三个问题的统一解法（Trinity）**

> **引用框**: "头疼可能需要治脚" — 当你从宏观视角出发，三个独立项目变成了一个统一框架。认识本质不只是看清单个问题，更是**看到不同问题之间的共性结构**。

> Speaker Notes: 再讲一个腾讯时期"认识本质"的案例。当时团队同时在做三个独立项目 — 多兴趣建模、长期兴趣建模、长尾兴趣探索。三个小组各自迭代，进展缓慢。直到退一步看，发现它们本质上是同一个问题 — 用户偏好在 cluster 空间上的分布。多兴趣就是取直方图 top-N，长尾就是热度消偏，长期兴趣就是老种子配公平 embedding。一套聚类加一个直方图加定制化策略，三个问题就有了统一的解法，这就是 Trinity 框架。这个案例体现了"认识本质"的更高层应用 — 不只是看清单个问题的本质，更是看到不同问题之间的共性结构。正如 thoughts on work 里写的：头疼可能需要治脚，从宏观视角出发解决问题。多了解其他方向的核心逻辑，让它们在头脑中形成思维框架，说不定在未来某个瞬间带给你灵机一动的灵感。

### 故事回溯 — 从"交付任务"到"交付价值"

4. **符合预期则加大投入，否则及时止损** — 阶段验证达标 → 快速拿收益；阶段验证不达标 → 降低风险，不沉没成本

> **引用框**: 项目要和业务目标对齐，更要有明确的关键路径和阶段性验证。**符合阶段预期则加大投入快速拿收益，否则及时止损降低风险。**

## Page 28 — 2.3 故事回溯：偏好度建模的假设-验证循环

[type: content]

### 故事回溯 — 一个更深度的假设-验证案例

**问题出发**：推荐系统的标准范式是预估用户各种行为（点击、完播、互动）的概率，然后多目标融合最大化观看时长/VV。但这和真正目标"优化用户观看体验、提升留存"之间存在 gap — **用户可能看了很久但并不喜欢**

**核心假设**：能不能直接建模用户的"真实偏好度"，而不是用行为代理指标拼凑？

**三轮假设-验证**：

- **第一轮**：直接用多目标融合分数作为偏好度 → 本质上还是行为预估的加权，没有跳出旧范式
- **第二轮**：构建 teacher 网络（偏好度质量模型），用全量特征和后验信息生成 soft label → teacher 网络本身怎么定义"偏好度"成了新问题
- **第三轮**：引入弱监督学习 — teacher 网络综合多维度行为信号生成连续偏好度分数，student 网络通过 PFD 知识蒸馏学习 soft label → **最终推全，核心指标显著正向**

> **引用框**: 这个项目的价值不在于某个技巧，而在于**从问题本质出发，质疑了旧范式本身**，经过多轮假设-验证，找到了跳出旧框架的建模方式。

> Speaker Notes: 再讲一个更有深度的假设-验证案例 — 偏好度建模。推荐系统的标准范式是预估用户各种行为的概率然后多目标融合，最大化观看时长或 VV。但我们发现这和真正目标"优化用户体验"之间存在 gap — 用户可能看了很久但并不喜欢。于是提出假设：能不能直接建模用户的真实偏好度？第一轮尝试用多目标融合分数作为偏好度，发现本质还是行为预估的加权，没跳出旧范式。第二轮构建 teacher 网络生成 soft label，但 teacher 本身怎么定义偏好度成了新问题。第三轮引入弱监督学习，teacher 综合多维度信号生成连续偏好度分数，student 通过知识蒸馏学习，最终推全成功。这个项目的价值在于它不是在旧框架内优化参数，而是质疑了框架本身。每一轮失败都缩小了搜索空间，最终收敛到正确方案。
